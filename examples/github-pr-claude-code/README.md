# GitHub Pull Requests â†’ reviewer agent

Label a GitHub pull request with `conveyor:review` and OpenConveyor posts a
review comment generated by Claude Code.

The reviewer agent never pushes commits and never opens PRs. Its only side
effect on the git host is `gh pr review --comment`.

## Flow

1. Someone labels a PR on `acme/widgets` with `conveyor:review`.
2. GitHub POSTs to `https://<your-domain>/github/pull-request` with a
   `X-Hub-Signature-256` header.
3. The OpenConveyor trigger adapter validates the HMAC, filters to
   `action=labeled` + `label.name=conveyor:review`, and creates a `Task`
   in the `conveyor-tasks` namespace with the PR's URL as the prompt.
4. The `claude-code-reviewer` AgentClass runs. The agent fetches the diff
   with `gh pr diff`, hands it to `claude --print` along with the prompt,
   and posts the result back with `gh pr review --comment`.

### 2. Create the webhook signing Secret

If you already configured the GitHub Issues example, the same `github-webhook`
Secret works â€” GitHub signs all webhook events with the same key per repo.
Otherwise:

```bash
# 32 bytes of entropy is the documented minimum; see ADR-0006.
SECRET=$(openssl rand -hex 32)
kubectl -n conveyor-system create secret generic github-webhook \
  --from-literal=secret="$SECRET"
```

### 3. Apply the ClusterTriggerClass

```bash

```

### 4. Configure GitHub

- **Content type:** `application/json`
- **Secret:** the `$SECRET` from step 2
- **Events:** "Let me select individual events" â†’ `Pull requests` only

## Verify

```bash
# On any open PR:
gh pr edit <number> --repo acme/widgets --add-label conveyor:review

# Watch for the Task.
kubectl -n conveyor-tasks get tasks -w
```

Expected: within seconds, `kubectl get task` shows a `gh-pr-xxxxx` Task
transitioning through `Running` â†’ `Completed`, and a new review comment
appears on the PR. If `kubectl get tasks` is empty, check the trigger
adapter logs for signature errors.

## Security notes

- **The token must not have write access.** Issue a fine-grained PAT
  scoped to the target repo with `Pull requests: Read and write` (write
  is required to *post* a review) and no `Contents` access. The agent
  has no `git push` path, but the token is the only thing that prevents
  one from being added later.
- **`filters:` is doing real work.** Without `label.name=conveyor:review`
  every label edit on every PR becomes an agent run. Do not remove the
  filters "just for testing."
- The Task's egress allowlist is `api.github.com` + `github.com`. The
  Anthropic endpoint comes from `ClusterAgentClass.spec.requires.egress`.
- Re-labeling an already-reviewed PR will dispatch a second review. If
  this matters, gate via a workflow that removes the label after the
  first review or use a one-shot label (`conveyor:review-once`) you
  delete before re-applying.

## Customising the review

The default trigger maps `pull_request.html_url` into the prompt. To pass
a per-PR rubric, change the mapping to `pull_request.body` â€” the agent
still extracts the PR URL from the prompt with a regex, so any prompt that
mentions the PR URL works.

To switch from "comment" to "approve" or "request changes" reviews,
override `REVIEW_EVENT` on the agent container (requires customising the
AgentClass; not exposed via the Task spec today).
